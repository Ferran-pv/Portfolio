1 Main knowledge

1.1 First commit

add new branch: https://github.com/Ferran-pv/sp-scripts -> clicar a “main” (nom default branch)

git config --global user.email "ferranpauls10@gmail.com"
git config --global user.name "Ferran-pv"
#
git init
git remote add origin https://github.com/Ferran-pv/sp-scripts.git
cd /home/pferran/scripts
git add .
git commit -m "Initial commit" 
git push -u origin main # Careful: name of branch will be 'main'.

1.2 normal push

cd /home/pferran/scripts
git add . 
git commit -m backup
git push -u origin main

2 Useful stuff

2.1 Clone repository

2.2 Reversing changes

reset = local, revert = remote.

git reset --hard HEAD~1 # 1 commit before head
git revert HEAD

2.3 Fetch/pull

git fetch
git merge origin main

# or, same thing
git pull

2.4 Branching

2.4.1 New branch & change to it

git branch <name>
git switch <name> # or checkout

2.4.2 Branch union

2 options: merge & rebase.

git switch X # change to branch created to maintain the other stable
git commit # save changes
git switch main 
git commit
# We are in 'main' branch.

########## merge
git merge bugFix # update main branch with changes.

########## rebase
git switch X
gitrebase main

3 Definitions

3.1 Main knowledge

add: takes a modified file in your working directory and places the modified version in a staging area (tracked files).

commit: records a snapshot of all the (tracked) files in your directory. It compresses a commit as a set of changes from one version of the local repository to the next.

head: A head is simply a reference to a commit object. At any given time, one head is selected as the ‘current head’ (i.e. latest commit in the branch you are in). This head is aliased to HEAD.

fetch: download non-updated changes from a remote repository.

  • Doesn’t change your local
    .

pull: fetches remote changes and then merges them.

push: uploading your changes to a specified remote (“publish” your work).

3.2 Branching

merge: To include the changes from one branch to the other branch.

  • History is mantained.
  • If you do it multiple times, the final squeme is messy.

rebase: logs are modified after .

  • Goal:make logs of repository history look linear.
  • Problem: we cannot track when & how the commits were merged.

3.3 Reversing changes

reset: will move a branch backwards as if the commit had never been made. Only in your local repository, not commited changes.

revert: reverse changes and share those reversed changes with others (i.e. applies to commited changes).
